Guide Approach

This guide follows a management-first approach. All licensing, device groups, templates / folders, and security policy are configured on your management platform — Panorama or Strata Cloud Manager (SCM) — before any Azure infrastructure is built. This ensures firewalls bootstrap into a fully configured management plane on first boot.

Design model: Hub-and-spoke with Azure Gateway Load Balancer (GWLB) and Internal Load Balancer for east-west inspection. Phase 2 lets you pick the management platform that fits your environment.

Deployment method: All Azure infrastructure (Phases 4–6) should be deployed with Terraform using the official Palo Alto Networks SWFW Modules for Azure. These modules encode Palo Alto's reference architecture and handle VNet, GWLB, route tables, and VM-Series provisioning with tested, production-ready code.

Architecture Overview

Before deploying anything, study the target architecture. This guide covers a centralized hub-and-spoke design: a Transit VNet hosts the VM-Series firewalls, and application workloads live in Spoke VNets that connect to the hub via VNet peering. All inbound, outbound, east-west, and on-prem traffic is inspected by the firewalls in the Transit VNet.

The diagrams below are organized into four groups: deployment models (the two main firewall layout choices), the Hub VNet, the Spoke VNet, and end-to-end traffic flows.

Each diagram below is collapsed by default — expand the ones you need.

Deployment Models

There are two layout choices for how VM-Series firewalls are organized inside the Transit VNet. Pick one before deploying — the choice affects the firewall count, traffic isolation between inbound and OBEW (Outbound + East-West) flows, and how you scale.

▶ Common Model

A single set of VM-Series firewalls handles all traffic flows — inbound, outbound, east-west, and backhaul to on-prem. The firewalls share UNTRUST and TRUST subnets, sit behind a public Azure Load Balancer (or Application Gateway) on the UNTRUST side, and behind an internal Azure Load Balancer on the TRUST side.

When to use: proof-of-concepts, smaller-scale deployments, or environments where operational simplicity matters more than per-flow isolation. Lower firewall count = lower cost, but inbound and outbound traffic share the same dataplane.

Tradeoff: all flows share one firewall pool. A bandwidth spike in one direction (e.g. heavy inbound) can degrade the others. Scaling means scaling the entire pool together.

▶ Transit Dedicated

Two separate sets of VM-Series firewalls inside the same Transit VNet:

  • Inbound tier — dedicated to internet-to-application traffic only.
  • OBEW tier — dedicated to outbound, east-west, and backhaul (on-prem) traffic.

Each tier has its own UNTRUST/TRUST interfaces and its own load balancers. Because the tiers are isolated at the dataplane level, an inbound bandwidth spike doesn't affect outbound or east-west performance.

When to use: production deployments where inbound and outbound have different scale or performance characteristics, or where you want to size each tier independently. Higher firewall count (typically 4 instead of 2) but each tier scales on its own.

Inbound load balancer choice: the inbound tier can sit behind either a public Standard Load Balancer (NLB) for simple L4 inbound, or an Application Gateway (AppGW) when you need L7 features (WAF, host-based routing, TLS offload, XFF header injection).

When to choose AppGW over NLB

Use Application Gateway when you need any of: web application firewall (WAF), TLS offload, host-based or path-based routing, X-Forwarded-For header insertion, or end-to-end TLS to multi-site backends. Hub VNet (Transit)

The Transit VNet hosts the VM-Series firewalls and the load balancers that front them. Every traffic flow in this design ultimately passes through here for inspection.

▶ Transit VNet — Subnet Layout

The Transit VNet uses the address space 10.51.0.0/24, divided into five /27 subnets:

SubnetCIDRPurpose
untrust-lb10.51.0.0/27Public Load Balancer / Application Gateway frontend
untrust10.51.0.32/27VM-Series untrust interface (eth1/1)
management10.51.0.64/27VM-Series management interface (eth1/0)
trust10.51.0.96/27VM-Series trust interface (eth1/2) and internal TRUST Load Balancer
gateway10.51.0.128/27Azure Virtual Network Gateway (for backhaul/VPN to on-prem)

VM-Series firewalls are deployed across Availability Zones (typically zones 1 and 2) for zone-redundant HA. Each firewall has three NICs: management, untrust, and trust.

Management subnet placement

This reference layout places the VM-Series management interface inside the Transit VNet's management subnet, while a separate Management VNet at 10.52.0.0/24 hosts the Panorama HA pair and peers with Transit. This isolates the Panorama management plane from data-plane subnets while keeping the firewall mgmt NIC adjacent to the data NICs for simpler routing.

▶ Transit VNet — Route Tables (UDRs)

Blackhole routes (next_hop_type = None) are added on each firewall subnet pointing to the other firewall subnets — this prevents traffic from short-circuiting between subnets without going through the firewall dataplane.

TRUST subnet UDR points at the firewall, not the ILB

The TRUST subnet's default route (0.0.0.0/0) points to a VM-Series trust interface IP, not to the internal Load Balancer. Spoke VNets and the on-prem gateway send traffic to the ILB; the firewall returns inspected traffic, and the TRUST UDR forwards it on. Pointing the TRUST UDR at the ILB itself would create a routing loop.

Spoke VNet

Spoke VNets host the actual application workloads. They peer with the Transit VNet and route all egress through the firewall TRUST internal Load Balancer.

▶ Spoke VNet — Topology

Each application gets its own Spoke VNet with a dedicated address space, peered with the Transit VNet. Example layout:

Spoke VNetAddress SpaceWorkload Subnet
App0110.61.1.0/2410.61.1.0/26 (with internal Application LB at 10.61.1.100)
App0210.61.2.0/2410.61.2.0/26

A separate Management VNet at 10.52.0.0/24 hosts the Panorama HA pair (10.52.0.4 primary, 10.52.0.5 secondary) and peers with Transit.

VNet peering is non-transitive

Spoke-to-spoke traffic does not automatically traverse the hub. To force spoke-to-spoke through the firewall, each spoke needs a UDR pointing remote spoke CIDRs (or a default route) at the TRUST internal Load Balancer. Without this, peered spokes would route directly to each other (if their VNets were also peered) or have no path at all.

▶ Spoke VNet — Route Tables (UDRs)

Each spoke workload subnet has a UDR that sends all non-local traffic to the TRUST internal Load Balancer in the Transit VNet (10.51.0.100 in the reference layout). This forces inspection on every flow leaving the spoke.

Route TableAssociated SubnetDestinationNext Hop
spoke-app-rtApp workload subnet (e.g. 10.61.1.0/24)0.0.0.0/0VirtualAppliance → 10.51.0.100 (TRUST ILB)

Outbound, east-west, and backhaul traffic all use this same default route. Inbound traffic arrives at the workload via the public Load Balancer / AppGW backend pool path, not via UDR.

Traffic Flows

Five end-to-end flows show how packets traverse the inspection layer in each direction. The flows assume the Common Model layout; the Transit Dedicated model splits the firewall pair (inbound flow uses the dedicated inbound pair; outbound, east-west, backhaul, and management use the OBEW pair) but the route logic is otherwise identical.

▶ Inbound — Internet to Spoke Application

Internet client → Application Gateway (or public LB) → VM-Series untrust interface → VM-Series trust interface → spoke workload via VNet peering. Source/Destination NAT happens on the firewall so the spoke workload sees the original client source.

▶ Outbound — Spoke Application to Internet

Spoke workload → spoke UDR 0.0.0.0/0 → TRUST internal Load Balancer (10.51.0.100) → VM-Series trust → VM-Series untrust → public Load Balancer outbound rule (or firewall public IP) → internet. The firewall source-NATs to its own untrust IP on the way out.

▶ East-West — Spoke to Spoke

App01 workload → spoke UDR sends remote spoke CIDR to TRUST ILB → VM-Series trust → policy inspection → VM-Series trust → App02 workload via VNet peering. Both halves of the round-trip traverse the firewall.

VNet peering between spokes is not required

You don't need to peer spoke VNets to each other — the hub-spoke peering plus UDRs steer all spoke-to-spoke traffic through the firewall. Direct spoke-to-spoke peering would actually break inspection by giving traffic a more-specific path that bypasses the hub.

▶ Backhaul — Spoke to On-Prem

Spoke workload → spoke UDR → TRUST ILB → VM-Series → Gateway subnet (Azure Virtual Network Gateway) → ExpressRoute or VPN tunnel → on-premises network. The Gateway subnet has its own UDRs sending on-prem prefixes back through the TRUST ILB so return traffic is also inspected.

▶ Management — Panorama and Admin Access

The VM-Series management interface (eth1/0, on the management subnet) registers to Panorama. Admin access reaches the management interface either directly (with a public IP and NSG allowlist) or via a jump host / Bastion in a peered Management VNet. The management plane is independent from the data plane — management traffic does not traverse the data plane interfaces.

▶ 1.4 Create a Deployment Profile

Palo Alto Docs: Create a Deployment Profile | Manage a Deployment Profile

The deployment profile defines the firewall model, vCPU allocation, and subscriptions that each bootstrapped firewall will receive.

  1. In the Customer Support Portal, go to Products > Software NGFW Credits.
  2. Select your credit pool and click Create Deployment Profile.
  3. Configure: Profile Name, Firewall Model, vCPU Count, Subscriptions, Token Count.
  4. Click Create. Note the generated auth code (format: D1234567).
Profiles partition by licensing tier and vCPU, not by topology

A deployment profile is a license SKU bundle — firewall model, vCPU count, and subscription mix (Threat Prevention, WildFire, URL Filtering, etc.). The auth code the profile generates is what each bootstrapping VM-Series presents to license itself.

The profile is not a grouping mechanism for region, cloud, function, or topology. Two firewalls in different clouds can share one profile if they have the same shape; two firewalls in the same VNet can need different profiles if their vCPU counts or subscription tiers differ.

Rule of thumb: one profile per distinct (vCPU count + subscription bundle). A few common patterns:

  • All firewalls identical → one profile, one auth code.
  • Inbound at 4 vCPU, OBEW at 8 vCPU → two profiles (different vCPU counts).
  • Inbound with extra subscriptions (e.g. Advanced WildFire), internal-only with the base bundle → two profiles (different subscriptions) even if the vCPU counts match.
  • Same shape deployed across AWS, Azure, and GCP → one profile shared across all three clouds.

Plan your profile count by walking your firewall inventory and grouping by (vCPU count, subscription mix). The number of distinct combinations is the number of profiles — and auth codes — you need.

▶ 1.8 Azure Subscription, RBAC & Service Principal for Terraform

Terraform requires an Azure identity with sufficient privileges to provision all resources (resource groups, VNets, NSGs, route tables, load balancers, public IPs, VMs, storage accounts). You have two options:

Option A: Logged-in user (development / Cloud Shell)

The simplest path for development is to authenticate Terraform with your own user account via az login. The user must have Contributor on the target subscription (or scoped to a resource group).

az login
az account set --subscription "<your-subscription-id>"
az account show --query '{subscription:name, id:id, tenantId:tenantId}'

Option B: Service Principal (Preferred for CI/CD)

Create a Service Principal scoped to the target subscription. This provides a non-interactive identity for automation pipelines.

# Capture your subscription ID first
SUBSCRIPTION_ID=$(az account show --query id -o tsv)

# Create the SP with Contributor on the subscription
az ad sp create-for-rbac \
  --name "terraform-vmseries" \
  --role "Contributor" \
  --scopes "/subscriptions/${SUBSCRIPTION_ID}"

The command outputs appId, password, and tenant. Export them as environment variables for the AzureRM provider:

export ARM_CLIENT_ID="<appId>"
export ARM_CLIENT_SECRET="<password>"
export ARM_TENANT_ID="<tenant>"
export ARM_SUBSCRIPTION_ID="<subscriptionId>"

Verify vCPU Quota in Your Region

VM-Series typically runs on the Standard_DS3_v2 family or higher (4 vCPUs each). Two firewalls in Common Model = 8 vCPUs minimum; four firewalls in Transit Dedicated = 16 vCPUs. Confirm you have quota in your target region:

# Replace 'eastus' with your target region
az vm list-usage --location eastus --output table | grep -i 'standardDSv2'

If CurrentValue + new VMs exceeds Limit, request a quota increase via Subscription > Usage + quotas in the Azure Portal.

Important

The initial deployment requires Contributor or equivalent on the subscription due to the wide range of resources created. If you also need Terraform to create role assignments or other Service Principals (e.g. for the Panorama Azure plugin), add User Access Administrator as well. Scope can be reduced post-deployment.

▶ 1.10 Terraform Remote State Backend

A remote state backend is critical for team collaboration, maintaining a single source of truth, and preventing concurrent modifications. Azure's azurerm backend uses a Storage Account container — state locking is provided by the native blob lease, so no separate locking table is required.

Create the Backend Resources

Create a small bootstrap resource group with a Storage Account and a container for Terraform state. The Storage Account name must be globally unique and 3–24 lowercase alphanumeric characters.

RG=tfstate-rg
LOCATION=eastus
SA=<your-unique-storage-account-name>
CONTAINER=tfstate

# Resource group
az group create --name $RG --location $LOCATION

# Storage account (versioned, TLS 1.2 only)
az storage account create \
  --name $SA \
  --resource-group $RG \
  --location $LOCATION \
  --sku Standard_LRS \
  --kind StorageV2 \
  --min-tls-version TLS1_2 \
  --allow-blob-public-access false

# Enable blob versioning for state history
az storage account blob-service-properties update \
  --account-name $SA \
  --resource-group $RG \
  --enable-versioning true

# Container for state
az storage container create --name $CONTAINER --account-name $SA

Configure the Backend (backend.tf)

terraform {
  backend "azurerm" {
    resource_group_name  = "tfstate-rg"
    storage_account_name = "<your-unique-storage-account-name>"
    container_name       = "tfstate"
    key                  = "pan-vmseries/azure/terraform.tfstate"
  }
}
Note

Place the state Storage Account in the same region as your deployment to minimize latency. Blob versioning provides state history; the native blob lease handles concurrent-modification locking automatically — no DynamoDB-equivalent is needed.

▶ 2A.3 Create Device Groups

Palo Alto Docs: Manage Device Groups

Use a parent/child device group hierarchy. Parents hold shared objects; children hold model-specific security and NAT policy. The hierarchy below covers both deployment models from the Architecture Overview:

  • DG-AZ — top-level Azure parent (shared objects across all Azure deployments)
  • DG-AZ-COMMON-MODEL — Common Model child (single firewall pair, all flows)
  • DG-AZ-TVNET — Transit VNet (Dedicated) parent
  • DG-AZ-TVNET-INBOUND-USE — Inbound dedicated tier, US-East region (Logical Router Engine)
  • DG-AZ-TVNET-OBEW-USE — OBEW dedicated tier, US-East region (Logical Router Engine)
  • DG-AZ-TVNET-INBOUND-ARE-USE — Inbound dedicated, US-East (Advanced Routing Engine)
  • DG-AZ-TVNET-OBEW-ARE-USE — OBEW dedicated, US-East (Advanced Routing Engine)

Create them top-down so each child can be assigned to its parent:

  1. Navigate to Panorama > Device Groups.
  2. Click Add. Set Name to DG-AZ. Click OK.
  3. Click Add. Set Name to DG-AZ-COMMON-MODEL, Parent Device Group to DG-AZ. Click OK.
  4. Click Add. Set Name to DG-AZ-TVNET, Parent to DG-AZ. Click OK.
  5. For each TVNet variant (INBOUND/OBEW × LRE/ARE), click Add and set Parent to DG-AZ-TVNET:
    • DG-AZ-TVNET-INBOUND-USE
    • DG-AZ-TVNET-OBEW-USE
    • DG-AZ-TVNET-INBOUND-ARE-USE
    • DG-AZ-TVNET-OBEW-ARE-USE
Naming Convention

Cloud prefix: AZ for Azure. Region suffix: USE for US-East (extend with USW, EUN, etc. as you deploy in other regions). Routing engine: ARE tag indicates the device group runs PAN-OS Advanced Routing Engine; absence means legacy Logical Router Engine.

Important
The Device Group must exist on Panorama before the firewall attempts to bootstrap. If the firewall cannot find its assigned Device Group, registration will fail.

Azure Health Probe IP 168.63.129.16

Azure load balancers (both public NLB and internal ILB) probe firewall health from the well-known IP 168.63.129.16. The firewall must (1) have a route back to this IP via the relevant subnet gateway, and (2) have a security rule allowing it — covered in 2A.11. Without the route, probes timeout and Azure marks the firewall unhealthy, dropping it from the LB pool.

Two VRs in Common Model — why?

The Common Model uses two virtual routers (AZ-UNTRUST-VR and AZ-TRUST-VR) so the routing tables for the inbound (untrust) side and the OBEW (trust) side are completely separate. This prevents accidental route leakage between zones and forces all traffic to pass through the firewall's policy engine when crossing zones (instead of being routed within a single VR). The TVNet model already gets this isolation by physical separation (two distinct firewall sets), so a single VR per firewall is sufficient.

Why DNAT rules are From/To = same zone

Azure's NAT rules in this design use pre-NAT zones: the From and To zones are evaluated before destination translation. The packet arrives on the UNTRUST/INTERNET interface destined for the firewall's own untrust IP (the LB frontend resolves to it via DNAT in Azure). The PAN-OS NAT rule then DNATs to the spoke workload. Even though the post-NAT path leads into TRUST, the rule's From/To are both UNTRUST/INTERNET because that's what the original packet header looks like. The matching security rule (in 2A.11) uses the post-NAT zone for the destination (TRUST), which is the standard PAN-OS pattern.

  • auth-key hasn't been rotated since you copied it (rotation invalidates older keys).
  • Azure outbound on TCP 443 is permitted from the firewall management subnet to updates.paloaltonetworks.com and the licensing endpoints — otherwise the licensing handshake will fail even if Panorama registration succeeds.

Why blackhole routes?

The blackhole UDRs (next_hop_type = "None") prevent direct subnet-to-subnet traffic inside the Transit VNet, forcing every flow that crosses zones to traverse a VM-Series firewall via the load balancer. Without these, Azure's default system route would let mgmt talk to trust directly — bypassing inspection.

Both firewall pairs share one Transit VNet and one set of subnets. The "dedicated" comes from the firewall role and bootstrap mapping: 2 firewalls bootstrap to BD-AZ-TVNET-INBOUND and join inbound load-balancer pools; 2 bootstrap to BD-AZ-TVNET-OBEW and provide HA-ports for spoke default routes.

Use the same vnets = { ... } block from 4A.2 — no changes required. The dedicated firewalls are added in 4B.5 to share these subnets.

Why no separate inbound/OBEW subnets?

Azure Standard Load Balancer backend pools are scoped to a NIC, not a subnet. The example puts all 4 firewalls' untrust NICs in untrust-snet and all 4 trust NICs in trust-snet; the public LB only health-probes the inbound NICs into its backend pool, and the private LB only health-probes the OBEW NICs. Subnet separation isn't required and would only complicate routing.

▶ 4B.3 Define Load Balancers (Per Tier)

Same shape as the Common example: one public LB (DNATs inbound traffic to the Inbound firewalls' untrust NICs only) and one private LB with HA-ports at 10.51.0.100 (sends spoke / mgmt-VNet trust traffic to the OBEW firewalls' trust NICs only).

Use the same load_balancers = { ... } block from 4A.3. The per-tier separation comes from which firewall NIC is added to which LB backend (configured in 4B.5 via load_balancer_key on each interface).

Inbound firewall NICs → public LB only; OBEW firewall NICs → private LB only

This is what makes the deployment "dedicated" — see 4B.5. If you accidentally add Inbound firewall NICs to the private LB, OBEW spoke traffic will start flowing through Inbound firewalls and break the tier isolation.

▶ Panorama Connectivity

Firewalls must be able to reach Panorama on TCP 3978 (management) and TCP 28443 (device certificate). The most common cause of registration failure is missing or misconfigured network connectivity between the firewall management subnet and Panorama.

Connectivity Checklist

  1. VNet peering: If Panorama is in a separate VNet, verify peering exists and shows Connected on both sides. This is the most common issue.
  2. ExpressRoute / VPN: If Panorama is on-premises, verify the ExpressRoute circuit or VPN tunnel is up and the management subnet has a route to Panorama's IP.
  3. NSG on management subnet: Must allow outbound TCP 3978 and 28443 to Panorama's IP. Must also allow the return traffic inbound.
  4. NSG on Panorama side: Must allow inbound TCP 3978 and 28443 from the firewall management subnet CIDR.
  5. Route table: The management subnet route table must have a route that reaches Panorama — do not route management traffic through the firewalls themselves (circular dependency).
Internet access for licensing

Firewalls also need outbound internet access from the management interface to reach updates.paloaltonetworks.com for device certificate registration and content updates. This can be provided via a NAT Gateway on the management subnet (preferred) or a public IP on the management NIC (not recommended — tighten the NSG if you must).

▶ NSG Blocking Management Access

Azure NSGs are stateful but have a default deny-all inbound rule. If you cannot reach the firewall management interface:

  1. Navigate to the management NIC → Effective security rules to see which rules are being evaluated.
  2. Verify the management NSG has explicit Allow rules for:
    • TCP 22 (SSH) and 443 (HTTPS) from your admin source CIDR
    • TCP 3978 and 28443 to/from Panorama's IP
  3. Check rule priority — a lower-numbered deny rule will override a higher-numbered allow rule.
  4. Check NSG flow logs (if enabled) to see whether traffic is being allowed or denied.

Symptoms

  • Security policies using DAGs are not matching traffic as expected

Resolution

Option 2: Unblock device push for each unconfigured plugin on each Panorama instance:

# To revert:
request plugins dau plugin-name <plugin-name> unblock-device-push no
Not persistent across reboots
