VM-Series Bootstrap Guide

End-to-end guide: from credit pool activation through verified bootstrap across AWS, Azure, and GCP

AWS Deployment Guide Azure Deployment Guide
Phase 1 Prerequisites Phase 2 Method 1 Phase 3 Method 2 Phase 4 Method 3/4 Phase 5 Verify
Guide Approach

This guide focuses on bootstrapping — getting a VM-Series firewall from first boot to operational with the correct licenses, Panorama connection, and baseline configuration. It covers four bootstrap methods across AWS, Azure, and GCP.

For full deployment guides that include infrastructure provisioning (VPCs, VNets, load balancers, Terraform), see the AWS VM-Series Deployment Guide or the Azure VM-Series Deployment Guide.

Method Overview

VM-Series firewalls support four bootstrap methods. Each method trades off automation, infrastructure requirements, and internet access needs. The table below compares them at a glance — pick the method that fits your environment, then follow the matching section.

Method Best For Cloud Storage Required Internet Required Panorama Required
1. Panorama Licensing Plugin Auto-scaling, production fleets No Panorama only Yes
2. Simple Bootstrap Quick one-off deployments No Yes (firewalls) Yes
3. Cloud Storage Basic Structured, version-controlled Yes Yes (firewalls) Yes
4. Cloud Storage Full Air-gapped, standalone firewalls Yes Yes (firewalls) No
Preferred
Method 1: Panorama Licensing Plugin — Panorama handles all licensing automatically. Firewalls do not need direct internet access. Licenses are reclaimed when instances terminate. Best for auto-scaling and production environments.
Standard
Method 2: Simple Bootstrap — Pass bootstrap parameters directly via user data (AWS), custom data (Azure), or instance metadata (GCP). No cloud storage infrastructure needed. Firewalls license themselves directly against updates.paloaltonetworks.com.
Storage
Method 3: Cloud Storage Basic — Bootstrap files (init-cfg.txt + authcodes) stored in S3, Azure File Share, or GCS. Provides version-controlled, auditable bootstrap packages without bundling PAN-OS software.
Full
Method 4: Cloud Storage Full — Complete bootstrap package including bootstrap.xml, PAN-OS software, and content updates. The firewall boots fully configured without needing Panorama. Use for standalone or air-gapped environments.

Shared Prerequisites

Complete these prerequisites before using any bootstrap method. Steps 1.1 through 1.6 are performed in the Palo Alto Networks Customer Support Portal and on Panorama. Steps 1.7 and 1.8 cover network connectivity requirements.

▶ Step 1.1: Activate Software NGFW Credit Pool

The credit pool funds all VM-Series license deployments. Each firewall consumes credits based on the vCPU count and subscriptions defined in its deployment profile.

Reference: Activate Credits

  1. Log in to the Palo Alto Networks Customer Support Portal.
  2. Navigate to Assets > Software NGFW Credits.
  3. Locate the credit pool authorization code from the purchase order.
  4. Click Activate and enter the auth code to fund the pool.
  5. Verify the credit pool shows available credits for the desired VM-Series model and subscription tier.
Note

Credit pools are shared across the account. A single pool can fund firewalls across AWS, Azure, and GCP.

Verification

The Software NGFW Credits page shows the credit pool with a positive available balance.

▶ Step 1.2: Create a Deployment Profile

The deployment profile defines the license shape — firewall model, vCPU count, and subscription bundle — that each bootstrapped firewall receives.

Reference: Create a Deployment Profile

  1. In the Customer Support Portal, navigate to Assets > Software NGFW Credits.
  2. Select the credit pool and click Create Deployment Profile.
  3. Configure the profile:
    FieldValue
    Profile NameDescriptive name (e.g., prod-4vcpu-full)
    Firewall ModelVM-Series model (e.g., VM-300, VM-500)
    vCPU CountNumber of vCPUs to allocate
    SubscriptionsThreat Prevention, WildFire, URL Filtering, DNS Security, etc.
    Token CountNumber of firewalls this profile can license
  4. Click Create.
  5. Copy the generated auth code (format: D1234567). This is needed for bootstrap definitions on Panorama.
Profiles partition by vCPU and subscriptions, not by topology

A deployment profile is a license SKU bundle. Two firewalls in different clouds can share one profile if they have the same vCPU count and subscription mix. Two firewalls in the same VPC need different profiles if their shapes differ. Plan one profile per distinct (vCPU count, subscription bundle) combination.

Verification

The deployment profile appears under Assets > Software NGFW Credits > Deployment Profiles with an auth code assigned.

▶ Step 1.3: Create a Device Group on Panorama

The Device Group holds the security policies, objects, and rules pushed to managed firewalls. It must exist on Panorama before any firewall attempts to bootstrap.

Reference: Manage Device Groups

  1. Log in to Panorama.
  2. Navigate to Panorama > Device Groups.
  3. Click Add.
  4. Set Name to a descriptive value (e.g., DG-PROD). This is the value used in dgname.
  5. Click OK.
  6. Commit to Panorama: Commit > Commit to Panorama.
Important

If a firewall bootstraps and cannot find its assigned Device Group on Panorama, registration fails silently. Always create the Device Group before launching firewall instances.

Verification

The Device Group appears in the Panorama > Device Groups list after commit.

▶ Step 1.4: Create a Template Stack on Panorama

The Template Stack holds network, device, and system-level configuration pushed to managed firewalls. The bootstrap parameter tplname refers to the Template Stack, not an individual Template.

Reference: Manage Templates and Template Stacks

  1. Navigate to Panorama > Templates.
  2. If no base template exists, click Add (not Add Stack), name it (e.g., TPL-BASE), and click OK.
  3. Click Add Stack.
  4. Set Name to a descriptive value (e.g., TS-PROD). This is the value used in tplname.
  5. Add the base template(s) to the stack.
  6. Click OK.
  7. Commit to Panorama.
Verification

The Template Stack appears in the Panorama > Templates list with its member templates shown.

▶ Step 1.5: Obtain the Auth Code for Bootstrap Definitions

The auth code links the deployment profile to the licensing plugin on Panorama.

Reference: Manage a Deployment Profile

  1. In the Customer Support Portal, navigate to Assets > Software NGFW Credits > Deployment Profiles.
  2. Locate the deployment profile created in Step 1.2.
  3. Copy the Auth Code (format: D1234567).
Verification

The auth code is copied and ready for use in the bootstrap definition or authcodes file.

▶ Step 1.6: Generate Auto-Registration PIN ID and PIN Value

The PIN ID and PIN Value enable the firewall to automatically fetch a device certificate from the Palo Alto Networks licensing server during first boot. Without a valid PIN, the firewall cannot authenticate with the licensing infrastructure.

Reference: Install a Device Certificate on the VM-Series Firewall

  1. Log in to the Customer Support Portal.
  2. Navigate to Products > VM-Series Auth-Codes.
  3. Click Generate OTP or Create Registration PIN.
  4. Copy both generated values:
    • PIN ID — UUID format (e.g., 81a75f2a-4de0-4f0e-959f-f31b5df06814)
    • PIN Value — hex string (e.g., 7059943c5565454db6b83cdcd305f338)

These map to the bootstrap parameters vm-series-auto-registration-pin-id and vm-series-auto-registration-pin-value.

Verification

Both the PIN ID (UUID format) and PIN Value (hex string) are copied and stored securely.

▶ Step 1.7: Establish Backhaul Connectivity to Panorama

Firewall management interfaces must have a network path to Panorama on ports 3978 and 28443. This is the single most common cause of bootstrap failure.

Critical: #1 cause of bootstrap failure

If the firewall management interface cannot reach Panorama, the bootstrap process completes but the firewall never registers, never receives configuration, and never activates its license. Verify end-to-end routing before launching any firewall instance.

Establish connectivity using one of these options per cloud:

CloudPanorama On-PremisesPanorama in Cloud
AWS Direct Connect, Site-to-Site VPN Transit Gateway, VPC Peering
Azure ExpressRoute, Site-to-Site VPN VNet Peering, Virtual WAN
GCP Cloud Interconnect, Cloud VPN VPC Peering, Shared VPC

Test connectivity from an instance in the firewall management subnet before deploying:

nc -zv <panorama-ip> 3978
nc -zv <panorama-ip> 28443
Verification

Both nc commands return Connection succeeded from a test instance in the firewall management subnet.

Open the required ports between firewall management subnets and Panorama. Apply these rules to every management subnet where firewalls will be deployed.

Security rules (AWS Security Groups, Azure NSGs, or GCP VPC firewall rules) are in place for all management subnets. Ports 3978, 28443, 443, and 22 are open in the required directions.

Method 1: Panorama Licensing Plugin

Preferred
Panorama handles all licensing automatically. Firewalls do not need direct internet access for licensing — only Panorama requires outbound HTTPS to updates.paloaltonetworks.com. Licenses are reclaimed automatically when instances terminate.
PAYG Exception

PAYG (pay-as-you-go) firewalls licensed through the cloud marketplace do not need the SW Firewall License plugin. PAYG firewalls are licensed directly through the marketplace. Skip to Method 2 for PAYG deployments.

▶ Step 1.1: Install the SW Firewall License Plugin

Reference: Use Panorama-Based Software Firewall License Management

  1. Log in to Panorama.
  2. Navigate to Panorama > Plugins.
  3. Click Check Now to refresh the plugin list.
  4. Locate sw_fw_license and click Download.
  5. Once downloaded, click Install.
  6. Verify the plugin appears under Panorama > SW Firewall License in the left navigation.

Requirements:

  • Panorama 10.0.0 or later
  • VM-Series firewalls running PAN-OS 9.1.0 or later
  • Licensing API key already configured on Panorama
  • Panorama must have internet access to updates.paloaltonetworks.com
Important: NSX and Azure plugin dependencies

If deploying VM-Series for VMware NSX, install the VM-Series plugin for NSX (vm_series) before installing sw_fw_license. If the Azure VM-Series plugin (azure) is installed alongside sw_fw_license, ensure it is fully configured and in a Registered or Success state. Unconfigured plugins can block IP-tag forwarding. See Troubleshooting: Plugin IP-Tag Issues.

Verification

Panorama > SW Firewall License appears in the left navigation with sub-menus for Bootstrap Definitions and License Managers.

▶ Step 1.2: Create a Bootstrap Definition

The bootstrap definition maps a deployment profile auth code to the licensing plugin. Create one definition per deployment profile.

  1. Navigate to Panorama > SW Firewall License > Bootstrap Definitions.
  2. Click Add.
  3. Set Name to a descriptive value (e.g., bsd-prod).
  4. Set Auth Code to the code from the deployment profile (format: D1234567).
  5. Click OK.
Verification

The bootstrap definition appears in the Bootstrap Definitions list with the correct auth code.

▶ Step 1.3: Create a License Manager

The license manager associates a Device Group, Template Stack, and bootstrap definition — and controls automatic license reclamation for terminated instances.

  1. Navigate to Panorama > SW Firewall License > License Managers.
  2. Click Add.
  3. Configure:
    FieldValue
    Namee.g., lm-prod
    Device GroupSelect the Device Group from Step 1.3
    Template StackSelect the Template Stack from Step 1.4
    Bootstrap DefinitionSelect the definition from Step 1.2
    Auto Deactivate1-2 hours for auto-scaling; Never for static fleets
  4. Click OK.
  5. Commit to Panorama.
Note

The Auto Deactivate timer controls how long Panorama waits before reclaiming a license from a disconnected firewall. Setting it too short may cause premature reclamation during transient network issues.

Verification

The license manager appears in the list with the correct Device Group, Template Stack, and bootstrap definition assigned.

▶ Step 1.4: Collect Bootstrap Parameters

The license manager generates the bootstrap parameters that each firewall needs at launch.

  1. In the License Managers list, locate the license manager created in Step 1.3.
  2. In the Action column, click Show Bootstrap Parameters.
  3. Copy the displayed values:
    • auth-key — the plugin-generated authentication key
    • plugin-op-commands — starts with panorama-licensing-mode-on
Important: auth-key vs vm-auth-key

The auth-key from the licensing plugin is different from a manually created vm-auth-key (request bootstrap vm-auth-key generate). Do not confuse the two. Method 1 uses auth-key; Method 2 uses vm-auth-key.

Important: plugin-op-commands for PAN-OS 11.0+

For PAN-OS 11.0 and later, include advance-routing:enable in plugin-op-commands. The full value becomes advance-routing:enable,panorama-licensing-mode-on. On AWS with GWLB, also append aws-gwlb-inspect:enable and aws-gwlb-overlay-routing:enable.

Verification

The auth-key and plugin-op-commands values are copied and ready for use in the instance launch parameters.

▶ Step 1.5: Launch Instance with Bootstrap Parameters

Pass the bootstrap parameters to the VM-Series instance at launch time. The delivery mechanism differs per cloud.

▶ AWS — User Data / Terraform bootstrap_options

AWS passes bootstrap parameters via EC2 user data as a semicolon-delimited string. In Terraform, use the bootstrap_options map.

Terraform

bootstrap_options = {
  plugin-op-commands                    = "panorama-licensing-mode-on,aws-gwlb-inspect:enable,aws-gwlb-overlay-routing:enable"
  auth-key                              = "YOUR-AUTH-KEY-HERE"
  panorama-server                       = "YOUR-PANORAMA-IP"
  dgname                                = "YOUR-DG-NAME"
  tplname                               = "YOUR-STACK-NAME"
  mgmt-interface-swap                   = "enable"
  dhcp-send-hostname                    = "yes"
  dhcp-send-client-id                   = "yes"
  dhcp-accept-server-hostname           = "yes"
  dhcp-accept-server-domain             = "yes"
  vm-series-auto-registration-pin-id    = "YOUR-PIN-ID"
  vm-series-auto-registration-pin-value = "YOUR-PIN-VALUE"
}

User Data One-Liner

mgmt-interface-swap=enable;plugin-op-commands=panorama-licensing-mode-on,aws-gwlb-inspect:enable,aws-gwlb-overlay-routing:enable;auth-key=YOUR-AUTH-KEY-HERE;panorama-server=YOUR-PANORAMA-IP;dgname=YOUR-DG-NAME;tplname=YOUR-STACK-NAME;dhcp-send-hostname=yes;dhcp-send-client-id=yes;dhcp-accept-server-hostname=yes;dhcp-accept-server-domain=yes;vm-series-auto-registration-pin-id=YOUR-PIN-ID;vm-series-auto-registration-pin-value=YOUR-PIN-VALUE
AWS-Specific Parameters

mgmt-interface-swap=enable is required for GWLB deployments where the management interface is not eth0. Include aws-gwlb-inspect:enable and aws-gwlb-overlay-routing:enable in plugin-op-commands for GWLB integration.

Verification

The EC2 instance launches. After 5-10 minutes, the firewall appears in Panorama > Managed Devices > Summary.

Method 2: Simple Bootstrap

Standard
Quick bootstrap using a manually generated VM auth key and an auth code passed directly via user data / custom data / metadata. No cloud storage infrastructure needed. Firewalls must have outbound internet access to updates.paloaltonetworks.com for licensing.
▶ Step 2.1: Generate a VM Auth Key on Panorama

The VM auth key allows the firewall to register with Panorama without the licensing plugin.

  1. SSH into Panorama.
  2. Generate the key (valid for 1 year / max 8760 hours): 
    request bootstrap vm-auth-key generate lifetime 8760
  3. Copy the generated key.
  4. To view existing keys: 
    request bootstrap vm-auth-key show
Verification

The vm-auth-key show command displays the key with its expiration timestamp.

▶ Step 2.2: Launch Instance with Bootstrap Parameters

Pass the vm-auth-key and authcodes directly to the instance. Key differences from Method 1: use vm-auth-key instead of auth-key, include authcodes, and do not include panorama-licensing-mode-on in plugin-op-commands.

▶ AWS — User Data / Terraform

Terraform

bootstrap_options = {
  authcodes                             = "D1234567"
  vm-auth-key                           = "YOUR-VM-AUTH-KEY"
  panorama-server                       = "YOUR-PANORAMA-IP"
  dgname                                = "YOUR-DG-NAME"
  tplname                               = "YOUR-STACK-NAME"
  mgmt-interface-swap                   = "enable"
  plugin-op-commands                    = "aws-gwlb-inspect:enable,aws-gwlb-overlay-routing:enable"
  dhcp-send-hostname                    = "yes"
  dhcp-send-client-id                   = "yes"
  dhcp-accept-server-hostname           = "yes"
  dhcp-accept-server-domain             = "yes"
  vm-series-auto-registration-pin-id    = "YOUR-PIN-ID"
  vm-series-auto-registration-pin-value = "YOUR-PIN-VALUE"
}

Storage
Bootstrap files (init-cfg.txt + authcodes) stored in cloud storage (S3, Azure File Share, or GCS). Provides a version-controlled, auditable bootstrap package. Does not bundle PAN-OS software or content updates. Firewalls must have outbound internet access.
▶ Step 3.1: Create Cloud Storage

license/
plugins/
software/

▶ Azure — File Share (SMB)

▶ Step 3.2: Create init-cfg.txt

The init-cfg.txt file contains the bootstrap parameters that configure the firewall's management interface and Panorama connection on first boot.

type=dhcp
panorama-server=YOUR-PANORAMA-IP
panorama-server-2=YOUR-PANORAMA-IP-2
tplname=YOUR-STACK-NAME
dgname=YOUR-DG-NAME
vm-auth-key=YOUR-VM-AUTH-KEY
hostname=fw-prod-01
dns-primary=169.254.169.253
vm-series-auto-registration-pin-id=YOUR-PIN-ID
vm-series-auto-registration-pin-value=YOUR-PIN-VALUE
Syntax Rules
  • No spaces around =
  • One parameter per line
  • type is required (dhcp or static)
  • If type=static, also include ip-address, netmask, and default-gateway
  • No trailing whitespace or BOM characters — use a plain text editor
Cloud-native DNS servers

Use the cloud-native DNS resolver when no customer-preferred DNS exists: AWS 169.254.169.253, Azure 168.63.129.16, GCP 169.254.169.254.

Upload to storage:

# AWS
aws s3 cp init-cfg.txt s3://your-bootstrap-bucket/config/

# Azure
az storage file upload --share-name bootstrap --source init-cfg.txt --path config/init-cfg.txt --account-name yourstorageaccount

# GCP
gsutil cp init-cfg.txt gs://your-bootstrap-bucket/config/
Verification

The init-cfg.txt file exists in the config/ directory of the storage resource.

▶ Step 3.3: Create the authcodes File

Place a plain text file named authcodes in the license/ directory containing one auth code per line.

The file is named authcodes with no file extension — not authcodes.txt, not authcodes.key. The firewall expects this exact filename. The file must be plain text with no trailing whitespace or blank lines.

Method 4: Cloud Storage Full Bootstrap

Full
Complete bootstrap package including bootstrap.xml (full PAN-OS configuration), PAN-OS software images, and content updates. The firewall boots fully configured without needing Panorama. Use for standalone firewalls or air-gapped environments.
When to use Method 4 vs Methods 1-3

Method 4 is not needed when Panorama or Strata Cloud Manager manages the firewalls, since those platforms push configuration after bootstrap. Use Method 4 only for standalone firewalls or environments where the firewall must operate independently from first boot.

init-cfg.txt exists in config/ and authcodes exists in license/.

▶ Step 4.3: Create bootstrap.xml

The bootstrap.xml is a complete PAN-OS configuration that the firewall applies on first boot. Export the running configuration from a known-working reference firewall.

Reference: Create the bootstrap.xml File

Option A: Web UI Export

  1. Log in to a VM-Series firewall with the desired configuration.
  2. Navigate to Device > Setup > Operations.
  3. Click Export named configuration snapshot.
  4. Select running-config.xml and save.
  5. Rename the file to bootstrap.xml (case-sensitive).

Option B: API Export

# Generate an API key
curl -k "https://<firewall-ip>/api/?type=keygen&user=admin&password=<password>"

# Export the running configuration
curl -k "https://<firewall-ip>/api/?type=export&category=configuration&key=<api-key>" -o bootstrap.xml

Option C: CLI Export

scp export configuration from running-config.xml to <user>@<destination>:/path/bootstrap.xml
Review before deploying

When using a configuration from another firewall, these values will differ per environment and must be updated:

  • Management interface IP address, netmask, default gateway
  • Default routes and gateway IPs
  • Dataplane interface IP addresses
  • DNS server addresses
  • Hostname (must be unique per firewall)
  • Zone and interface mappings

Deploying with incorrect network values results in an unreachable firewall.

Upload to storage:

# AWS
aws s3 cp bootstrap.xml s3://your-bootstrap-bucket/config/

# Azure
az storage file upload --share-name bootstrap --source bootstrap.xml --path config/bootstrap.xml --account-name yourstorageaccount

# GCP
gsutil cp bootstrap.xml gs://your-bootstrap-bucket/config/
Verification

bootstrap.xml exists in the config/ directory alongside init-cfg.txt.

▶ Step 4.4: Upload Software and Content Files

Upload PAN-OS software images and content update files to their respective directories.

  • Software: Place PAN-OS image files in software/. Include all intermediate versions between the factory image and the target version. The firewall upgrades sequentially and fails if any version is skipped.
  • Content: Place the content update file in content/. The content version must meet the minimum required for the target PAN-OS version.
  • Plugins (optional): Place a single VM-Series plugin image in plugins/.
Content version compatibility

A content version mismatch causes the upgrade to fail. Check the PAN-OS release notes for the minimum required content version before uploading.

Verification

The software/ directory contains the target PAN-OS image. The content/ directory contains the compatible content update file.

Troubleshooting

▶ Bootstrap Failures — Firewall Stuck at 192.168.1.1

If the firewall management IP is 192.168.1.1, the bootstrap process failed and the firewall reverted to factory defaults.

CauseResolution
init-cfg.txt has spaces around =Remove all spaces around = signs. Use a plain text editor.
Storage is unreachable (IAM/access key/service account)Verify the instance profile, access key, or service account has read access to the storage resource.
Custom data / user data not passed at launchCheck the instance metadata in the cloud console to confirm bootstrap parameters were applied.
type parameter missing from init-cfg.txtAdd type=dhcp or type=static as the first line.
BOM characters in init-cfg.txtRe-save the file without BOM encoding. Use file init-cfg.txt to check.
Missing directories in storageAll five directories (config/, content/, license/, plugins/, software/) must exist.
Verification

After fixing the issue, terminate the instance and launch a new one. Check debug logview component bts_details for bootstrap progress.

The firewall boots but does not appear in Panorama > Managed Devices.

▶ Plugin IP-Tag Issues

Unconfigured plugins on Panorama can block IP-tag forwarding to managed firewalls, affecting Dynamic Address Group membership and security policy enforcement.

Workaround 1: Uninstall unconfigured plugins

  1. Navigate to Panorama > Plugins.
  2. Select the unconfigured plugin and click Uninstall.

Workaround 2: Unblock device push

Run this for each unconfigured plugin on each Panorama instance:

request plugins dau plugin-name <plugin-name> unblock-device-push yes

To reverse:

request plugins dau plugin-name <plugin-name> unblock-device-push no
Important

These commands are not persistent across reboots. Re-execute after any Panorama restart. For Panorama HA, execute on each instance.

Verification

All plugins under Panorama > Plugins show a Registered or Success state, or unconfigured plugins have been uninstalled.
