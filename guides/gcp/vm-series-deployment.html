VM-Series Deployment on GCP

End-to-end guide: Panorama configuration through GCP infrastructure to verified traffic inspection

Guide Approach

This guide follows a management-first approach. All licensing, device groups, templates, and security policy are configured on your management platform — Panorama or Strata Cloud Manager (SCM) — before any GCP infrastructure is built. This ensures firewalls bootstrap into a fully configured management plane on first boot.

Design model: VPC Peering hub-and-spoke with ILB sandwich architecture. Two topology options are documented: common-firewall (single firewall set for all traffic) and dedicated-inbound (separate inbound and outbound/east-west firewall sets). Phase 2 lets you pick the management platform that fits your environment.

Deployment method: All GCP infrastructure (Phases 4–6) is deployed with Terraform using the official Palo Alto Networks SWFW Modules for GCP. These modules encode the reference architecture and handle VPC, load balancer, VPC peering, and VM-Series provisioning with tested, production-ready code.

Architecture Overview

Before deploying anything, study the target architecture. GCP uses an ILB sandwich pattern for traffic inspection — fundamentally different from AWS (GWLB with Geneve encapsulation) and Azure (Gateway Load Balancer). Understanding this architecture is critical before configuring Panorama or writing Terraform.

GCP has no Gateway Load Balancer

Unlike AWS and Azure, GCP does not offer a Gateway Load Balancer. Traffic inspection uses an Internal Load Balancer (ILB) sandwich: an External NLB for inbound traffic and an Internal NLB for outbound and east-west traffic, with VM-Series firewalls as backends for both. Because the GCP ILB can only forward traffic to nic0 of backend instances, management interface swap is required — nic0 becomes the dataplane interface (ethernet1/1) and nic1 becomes the management interface.

Topology Options

The GCP Terraform modules support two primary topologies. Both use VPC Peering to connect spoke VPCs to a central trust VPC.

▶ Common-Firewall Topology

A single set of VM-Series firewalls handles all traffic flows: outbound, east-west, and inbound. The default deployment creates:

  • 5 VPC networks: Management (10.10.10.0/28), Untrust (10.10.11.0/28), Trust (10.10.12.0/28), Spoke-1 (192.168.1.0/28), Spoke-2 (192.168.2.0/28)
  • 2 VM-Series firewalls in separate zones (e.g., us-east1-b and us-east1-c)
  • 1 Internal NLB at 10.10.12.5 in the trust VPC (outbound + east-west traffic)
  • 1 External Regional NLB with a public IP (inbound traffic)
  • VPC Peering between trust VPC and each spoke VPC
  • 2 test Linux VMs in the spoke VPCs for connectivity testing

This topology is suitable for most deployments, proof-of-concept environments, and moderate traffic volumes. Approximately 96 Terraform resources are created.

Terraform example

examples/vpc_peering_common in the official module repository.

▶ Dedicated-Inbound Topology

Separates inbound traffic handling from outbound and east-west. The deployment creates:

  • Same 5 VPC networks as the common topology
  • 4 VM-Series firewalls: 2 for inbound (behind a Global HTTP Load Balancer) and 2 for outbound/east-west (behind the Internal NLB)
  • 1 Internal NLB in the trust VPC (outbound + east-west)
  • 1 Global HTTP Load Balancer with managed instance group backends (inbound)

This topology is appropriate for high-volume inbound traffic, production-scale deployments where inbound and outbound security policies diverge significantly, or when the inbound tier needs independent scaling. Approximately 104 Terraform resources are created.

Terraform example

examples/vpc_peering_dedicated in the official module repository.

▶ Traffic Flows

Outbound (Spoke to Internet)

A spoke VM sends traffic to the internet. The spoke VPC has its default internet route deleted; a custom route (0.0.0.0/0) with the ILB IP (10.10.12.5) as next-hop is imported via VPC Peering. Traffic flows:

  1. Spoke VM → VPC Peering → Trust VPC ILB (10.10.12.5)
  2. ILB selects a VM-Series backend (forwarding to nic0 = ethernet1/1 after mgmt-interface-swap)
  3. VM-Series inspects the traffic, applies security policy
  4. Post-inspection: firewall routes traffic out the untrust interface (ethernet1/1 or default route) to the internet via the Untrust VPC default route
  5. Return traffic follows the reverse path through the firewall

East-West (Spoke to Spoke)

A VM in Spoke-1 communicates with a VM in Spoke-2. Both spokes peer with the trust VPC:

  1. Spoke-1 VM → VPC Peering → Trust VPC ILB
  2. ILB → VM-Series (inspects with east-west policy)
  3. VM-Series routes the packet back through the trust interface to Spoke-2 via VPC Peering

Inbound (Internet to Spoke)

An external client connects to an application in a spoke VPC:

  1. External client → External NLB public IP
  2. External NLB → VM-Series nic0 (untrust/dataplane interface after swap)
  3. VM-Series inspects with inbound policy, then forwards to the spoke via the trust interface and VPC Peering
  4. Return traffic: spoke VM → ILB → VM-Series → External NLB → client
ILB sandwich architecture

The key architectural difference from AWS/Azure: there is no Geneve encapsulation or overlay. Traffic is forwarded at Layer 3 through the ILB to the VM-Series nic0 interface. The firewall performs standard routing between its trust and untrust interfaces. VPC Peering with exported custom routes provides the spoke-to-hub connectivity that AWS achieves with Transit Gateway.

Interface Mapping with Management Interface Swap

Management interface swap is required for any deployment using the Internal Load Balancer. The swap reassigns nic0 from management to dataplane.

PAN-OS InterfaceGCP NIC (Default)GCP NIC (After Swap)VPC Network
Managementnic0 (eth0)nic1 (eth1)fw-mgmt-vpc
ethernet1/1 (untrust)nic1 (eth1)nic0 (eth0) ← ILB sends herefw-untrust-vpc
ethernet1/2 (trust)nic2 (eth2)nic2 (eth2)fw-trust-vpc
Minimum 2 NICs required for swap

If a VM-Series instance is launched with only 1 network interface and mgmt-interface-swap is enabled, the firewall boots into maintenance mode. The Terraform modules create 3 NICs by default (untrust, management, trust) — do not reduce this below 2.

▶ 1.3 Generate SSH Key Pair

An SSH key is required for initial access to the VM-Series firewall management interface. GCP VM-Series instances accept exactly one key at deployment time.

ssh-keygen -t rsa -b 4096 -f ~/.ssh/gcp-vmseries-key -C "admin"

The public key must be formatted as: admin:ssh-rsa AAAA... admin

Both the admin: prefix and the trailing admin comment are mandatory. Display the correctly formatted key:

echo "admin:$(cat ~/.ssh/gcp-vmseries-key.pub)"
Corrupted key = delete and redeploy

If the SSH key is malformed (extra spaces, missing prefix/suffix, wrong algorithm), the firewall will deploy but SSH access will fail. The only recovery is to delete the VM-Series instance and redeploy. The VPC networks survive deletion — only the firewall instance and firewall rules must be recreated.

Verification

The output of echo "admin:$(cat ~/.ssh/gcp-vmseries-key.pub)" should start with admin:ssh-rsa AAAA and end with admin. No line breaks or extra whitespace.

Profiles partition by vCPU count and subscription bundle, not by topology

Windows Users

All shell commands in this guide use Bash syntax (Linux/macOS). If you are deploying from a Windows workstation, use one of the following approaches:

  • Windows Subsystem for Linux (WSL) — Recommended: Install WSL 2 with Ubuntu, then install Terraform, gcloud CLI, and Git inside the WSL environment. All commands in this guide work as-is.
  • Git Bash: Installed alongside Git for Windows. Provides a Bash-compatible shell where most commands work without modification.
  • PowerShell: Terraform and gcloud CLI work natively, but shell syntax differs — environment variables use $env:VAR instead of $VAR, line continuations use backtick (`) instead of backslash (\), and curl is aliased to Invoke-WebRequest. Use curl.exe or Invoke-WebRequest explicitly.

SSH Key Permissions on Windows

Windows does not honor Unix file permissions. If SSH refuses your key with "Permissions are too open", restrict the key file with:

icacls key.pem /inheritance:r
icacls key.pem /grant:r "%USERNAME%:(R)"

This removes inherited permissions and grants read-only access to your user account, which is the Windows equivalent of chmod 400.

Phase 2: Management Platform Configuration

VM-Series can be managed by either Panorama (on-premises or cloud-hosted management appliance) or Strata Cloud Manager (SCM) (SaaS management plane). Pick the path that matches your environment.

Which path should I pick?
  • Panorama — choose if you already operate Panorama in your environment, need on-premises control, or are using the SW Firewall License Plugin for licensing.
  • SCM — choose for greenfield deployments, cloud-native operations, or if consolidating onto Strata's SaaS management plane.

▶ 2A.8 Configure Management Interface Profile

The ILB and External NLB health checks probe the VM-Series on port 80 (HTTP) at the path /php/login.php. An interface management profile must allow HTTP on the dataplane interface that receives health check traffic.

  1. Navigate to Network > Network Profiles > Interface Mgmt.
  2. Click Add. Set Name to allow-health-check.
  3. Under Network Services, check HTTP (port 80).
  4. Click OK.
  5. Return to Network > Interfaces > ethernet1/1.
  6. Under Advanced > Other Info, set Management Profile to allow-health-check.
Important: Health check path

Both the ILB and External NLB use HTTP GET to /php/login.php on port 80. This is the default PAN-OS login page. If HTTP is not enabled on the interface management profile, health checks fail and the load balancers mark the firewall as unhealthy.

Verification

The management profile allow-health-check appears under Interface Mgmt with HTTP enabled, and is assigned to ethernet1/1.

2B SCM Path

For SCM-managed deployments, the GCP Terraform configuration works identically to the Panorama path — the bootstrap parameters differ but the infrastructure is the same.

▶ 2B.1 SCM Bootstrap Configuration

Instead of Panorama IP and VM auth key, provide SCM-specific bootstrap parameters in the Terraform bootstrap_options block:

bootstrap_options = {
  type                                  = "dhcp-client"
  mgmt-interface-swap                   = "enable"
  panorama-server                       = "cloud"
  dgname                                = "your-scm-folder-name"

Key differences from the Panorama path:

All other steps in this guide (VPC creation, ILB configuration, spoke peering) are identical regardless of management platform. Proceed to Phase 3 after configuring your SCM folder with zones, interfaces, security policy, and NAT policy matching the Panorama path above.

Verification

▶ 4.6 Configure VM-Series Bootstrap

Set the bootstrap options in the vmseries_common block. These parameters are passed to each firewall as instance metadata and control how the firewall configures itself on first boot.

vmseries_common = {
  ssh_keys            = "admin:ssh-rsa AAAA...your-key... admin"
  vmseries_image      = "vmseries-flex-byol-1110h3"
  machine_type        = "n2-standard-4"
  min_cpu_platform    = "Intel Cascade Lake"
  service_account_key = "sa-vmseries-01"
  bootstrap_options = {
    type                                  = "dhcp-client"
    mgmt-interface-swap                   = "enable"
    panorama-server                       = "10.255.0.4"
    panorama-server-2                     = "10.255.0.5"
    tplname                               = "STK-GCP-SECURITY"
    dgname                                = "DG-GCP-SECURITY"
    vm-auth-key                           = "your-vm-auth-key"
    authcodes                             = "D1234567"
    vm-series-auto-registration-pin-id    = "your-pin-id"
    vm-series-auto-registration-pin-value = "your-pin-value"
  }
}

Also set per-firewall DNS in the individual vmseries blocks:

vmseries = {
  fw-vmseries-01 = {
    # ...
    bootstrap_options = {
      dns-primary   = "169.254.169.254"
      dns-secondary = "8.8.8.8"
    }
    # ...
  }
}
GCP metadata DNS

169.254.169.254 is the GCP metadata server, which resolves internal DNS names and forwards to Google Public DNS. Use it as the primary DNS for simplest configuration. Substitute customer-preferred DNS servers if required by organizational policy.

The mgmt-interface-swap = "enable" parameter must be set in the bootstrap options. Without it, the ILB sends traffic to nic0 (management) instead of a dataplane interface, and all load-balanced traffic is silently dropped.

▶ Traffic Not Flowing Through Firewalls
  1. Check VPC routes: The trust VPC must have a route 0.0.0.0/0 → ILB. Spoke VPCs must have this route imported via peering.
  2. Check VPC peering: Both peerings must be Active with custom route exchange enabled.
  3. Check spoke default route deletion: Spoke VPCs must have delete_default_routes_on_create = true. If the Google default internet route still exists, it takes precedence and traffic bypasses the firewall.
  4. Check can_ip_forward: The VM-Series instances must have IP forwarding enabled. The Terraform module sets can_ip_forward = true by default.
  5. Check firewall routing: Run show routing route on the firewall CLI. Verify a default route via ethernet1/1 (untrust gateway) and routes to spoke subnets via ethernet1/2 (trust).
  6. Check security policy: Run show log traffic to see if sessions are being created. If traffic is denied, check the matching rule and adjust policy.
Verification

After resolving, spoke VMs can reach the internet (curl ifconfig.me returns an IP) and other spokes (ping 192.168.2.2 succeeds). Traffic logs appear on the firewall.

▶ Plugin IP-Tag Issues

If multiple plugins are installed on Panorama and one or more is unconfigured, IP-tag forwarding may be blocked.

Workaround 1

Uninstall unconfigured plugins: Panorama > Plugins > select plugin > Uninstall.

Workaround 2

Unblock device push:

request plugins dau plugin-name <plugin-name> unblock-device-push yes
Important

These commands are not persistent across reboots and must be re-executed after any Panorama restart. For HA pairs, run on each Panorama instance.

Verification

IP-tag information appears on the managed firewalls after the unblock command.
