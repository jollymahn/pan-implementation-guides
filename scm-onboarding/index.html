SCM Firewall Onboarding Guide

End-to-end procedure: prerequisites, device onboarding, and post-onboarding configuration

Audience Paths
  • NGFW (hardware firewalls) — Steps 1–4, then Step 6 (Onboarding) and Step 7 (Post-Onboarding)
  • VM-Series (Software NGFW Credits) — All 7 steps (Step 5 covers VM-Series specific prerequisites)
  • Prisma Access — Steps 1–2, then Step 6 Section 6.2
  • Panorama-managed firewalls — Steps 1–4 (use Section 4.2 for certificates); add Step 5 if VM-Series; Step 7 for CloudConnector
  • Note: Step 1 is hardware/software verification; Step 2 is licensing & accounts

Procedure Overview

Work through each step in order — each builds on the previous one. Complete every required item before moving to the next step.

  1. Step 1. Hardware & Software Compatibility — supported models, PAN-OS versions, single vsys
  2. Step 2. Licensing & Account Readiness — Activation Console, SCM license, device association, deployment profiles
  3. Step 3. Network & Connectivity — ports, FQDNs, DNS, NTP, proxy configuration
  4. Step 4. Device Certificates — install and verify certificates on all firewalls
  5. Step 5. VM-Series Specific Prerequisites — credit activation, deployment profiles, auto-registration PINs, cloud platform readiness (VM-Series only)
  6. Step 6. Onboard Devices to Strata Cloud Manager — configure firewalls for cloud management, associate in SCM, verify connectivity
  7. Step 7. Post-Onboarding Setup & Automation — CloudConnector plugin, device labels, onboarding rules, configuration push

2.2 Activate & Verify License Status

  1. Log in to the Activation Console (also referred to as “the Hub”).
  2. Navigate to Common Services > Tenant Management.
  3. Verify your SCM license tier (Essentials or Pro) is active.
  4. Check that the Activation Status shows Complete.
  5. If using SCM Pro, confirm Strata Logging Service shows as included.
  6. If using SCM Essentials and you need logging, activate the SLS add-on separately.
Wait for Activation to Complete

The Activation Status must show Complete before you proceed. License provisioning can take up to 30 minutes. If the status shows “In Progress” or “Pending,” wait for it to complete before continuing — services will not function until provisioning finishes.

2.3 Verify Deployment Profile Association (VM-Series only)

This section applies if you are deploying VM-Series firewalls using Software NGFW Credits. It assumes you have already activated a credit pool and created a deployment profile in the Customer Support Portal. If you have not yet created a deployment profile, see Create a Deployment Profile first.

Once your SCM license activation is complete, verify that your deployment profile is associated with the correct tenant:

  1. In the Activation Console, navigate to Common Services > Tenant Management > Deployment Profiles.
  2. Locate your deployment profile (search by name or auth code starting with “D”).
  3. Check the Profile Association Status:
    • If it shows Complete — your profile is correctly associated. Proceed to Section 2.4.
    • If it shows Associating — provisioning is in progress. Wait up to 50 minutes for it to complete.
    • If your deployment profile is not listed — it has not been associated with this tenant yet. Follow the steps below.

Associate a Deployment Profile with Your Tenant

If your deployment profile does not appear under the tenant’s Deployment Profiles tab:

  1. Navigate to Common Services > Subscriptions & Add-ons.
  2. Look for your deployment profile under Ready for Activation — click Activate Now.
    Alternatively: In CSP under Software NGFW Credits > Current Deployment Profiles, click Finish SetUp next to the profile — this redirects you to the Activation Console.
  3. Authenticate with SSO if prompted.
  4. On the Activate Subscriptions based on Deployment Profiles page:
    • Select the CSP account you are subscribed to.
    • Select (or create) the target tenant and region.
    • Select the deployment profile to associate (search by name or auth code).
  5. (Optional) Select Additional Services such as Cloud Identity Engine.
  6. Agree to Terms and Conditions → click Activate.
  7. Wait for the Profile Association Status to show Complete (provisioning takes approximately 15–50 minutes).
Consistency Rule

A deployment profile can only be associated with one Tenant Service Group (TSG) at a time. All deployment profiles linked to the same TSG must use the same management subscription. You cannot mix SCM Pro and Strata Logging Service on the same tenant.

Reference: Create a Deployment Profile | Activate the Deployment Profile

VM-Series: Not Yet Deployed?

VM-Series firewalls do not have serial numbers until they are deployed, licensed, and bootstrapped. This is why the deployment profile is important — the auth code from your deployment profile must be included in the bootstrap data so the firewall can license and register itself automatically.

If your VM-Series firewalls have not been deployed yet, skip this step for now — you will associate them after deployment. VM-Series firewalls using Software NGFW Credits register automatically when they boot with a valid auth code and auto-registration PIN, so they will appear in Device Associations once deployed. Hardware NGFWs should be associated at this stage.

See Step 5: VM-Series Specific Prerequisites for bootstrap configuration details, including how to include the auth code and auto-registration PIN in your init-cfg.txt or cloud instance user data.

2.4 Associate Devices to Tenant

  1. In the Activation Console, navigate to Common Services > Device Associations.
  2. Click Add Devices.
  3. Select all firewalls to be onboarded by their serial numbers.
  4. Confirm each device has an active support license.
  5. Click Save.
Note

Devices must have a valid device certificate installed before they will associate with the correct tenant and can be onboarded to SCM.

  • Hardware NGFWs: Generate an OTP and install the certificate manually or via Panorama. See Step 4: Install Device Certificates. You also need a Customer Support Portal (CSP) account with Super User or Standard User role for OTP generation.
  • VM-Series: The device certificate is installed automatically during bootstrap when you include the auto-registration PIN ID and PIN Value in your init-cfg.txt or cloud instance user data. This is why the PIN is a required element in the SCM bootstrap process. See Step 5, Section 5.4: Generate Auto-Registration PIN for details.
▶ 2.5 Prisma Access Requirements (if applicable)

  • Panorama version 10.0 or higher (if currently Panorama-managed)
  • Account with superuser privileges

3.7 Proxy Considerations

Proxy Requirements

If outbound traffic routes through a proxy:

  • The proxy must support non-standard SSL ports TCP 3978 and TCP 444
  • CloudConnector plugin 2.2.0+ supports proxy settings configured in Panorama (requires a commit after configuration; settings take effect on the next commit cycle)

3.8 Configure DNS & NTP

Both DNS and NTP must be configured on every firewall before onboarding. NTP is required for certificate validation.

  1. Log into the firewall web interface.
  2. Navigate to Device > Setup > Services → edit Services.
  3. Configure DNS — enter Primary and Secondary DNS server addresses.
  4. Select NTP → enter Primary NTP server (and optional Secondary).
  5. Click OK → Commit.
  6. Verify NTP sync:
admin> show ntp

Expected result: status shows synced.

3.9 Validate Connectivity

Run these commands from each firewall CLI to confirm network readiness:

# Test DNS resolution to SCM discovery service
admin> ping host ds-cloudmgmt.paloaltonetworks.com

# Test DNS resolution to certificate services
admin> ping host certificate.paloaltonetworks.com

# Verify NTP sync
admin> show ntp

# Check license status
admin> request license info

Step 4: Install Device Certificates

Device certificates authenticate your firewalls with Palo Alto cloud services. This must be done before onboarding.

Auto-Install Models

PA-400, PA-1400, PA-3400, PA-5400, PA-5450, PA-5500, and PA-7500 series firewalls install the certificate automatically during initial registration. Verify the certificate is valid; if not, follow the manual process below.

Certificate Lifetime

Certificates are valid for 90 days with automatic renewal 15 days before expiration.

▶ 4.1 Standalone Firewalls: Generate OTP & Install Certificate

Palo Alto Docs: Install Device Certificate

Use this workflow when the firewall is not managed by Panorama.

Step 1: Generate a One-Time Password (OTP)

Important

The OTP is valid for 60 minutes and allows only one attempt. Have the firewall ready before generating.

  1. Log into the Customer Support Portal.
  2. Navigate to Products > Device Certificates.
  3. Click Generate OTP.
  4. Select Generate OTP for a Next-Gen Firewall (PAN-OS) → Next.
  5. Select your device serial number → Generate OTP.
  6. Download or Copy the OTP immediately.

Step 2: Install the Certificate

Option A — Web Interface:

  1. Log into the firewall web interface as Superuser.
  2. Navigate to Device > Setup > Management > Device Certificate.
  3. Click Get certificate → paste the OTP → OK.

Option B — CLI:

admin> request certificate fetch otp

Step 3: Verify

Refresh the page and confirm Current Device Certificate Status shows as Valid with a future expiration date.

admin> show device-certificate status
▶ 4.2 Panorama-Managed Firewalls: Install Certificates via Panorama

Palo Alto Docs: Install Device Certificate for Managed Firewalls

Use this workflow when firewalls are managed by Panorama. This lets you install certificates on multiple firewalls at once without logging into each one individually.

Prerequisites

  • Panorama role: Superuser (required to generate the OTP Request Token)
  • Licenses: Device Management license and Support license on Panorama
  • CSP alignment: Managed firewalls must belong to the same CSP account as Panorama
  • Outbound internet access from each managed firewall

Step 1: Configure NTP on Panorama and Managed Firewalls

  1. On Panorama: Go to Panorama > Setup > Services → select NTP → enter the Primary NTP server → OK → Commit to Panorama.
  2. On managed firewalls (via Template): Go to Device > Setup > Services and select the appropriate Template → enter NTP server details → OK → Commit and Push.

Step 2: Generate the OTP Request Token on Panorama

  1. Log into Panorama as a Superuser.
  2. Navigate to Panorama > Managed Devices > Summary.
  3. Select the managed firewalls that require a certificate.
  4. Select Request OTP From CSP > Custom selected devices.
  5. Copy the entire output in the OTP Request Token field.

Step 3: Generate the OTP on CSP

Important

The OTP is valid for 60 minutes and allows only one attempt.

  1. Log into the Customer Support Portal.
  2. Navigate to Products > Device Certificates > Generate OTP.
  3. Select Generate OTP for Panorama managed firewalls → Next.
  4. Paste the OTP Request Token copied from Panorama → Generate OTP.
  5. Wait a few minutes, then go to OTP History → Copy to Clipboard.

Step 4: Upload OTP and Install

  1. In Panorama, go to Panorama > Managed Devices > Summary.
  2. Click Upload OTP.
  3. Paste the OTP from CSP and click Upload.
Paste Only

You must paste the OTP text — uploading a downloaded file is not supported.

Step 5: Verify

In Panorama, navigate to Panorama > Managed Devices > Summary and confirm:

  • The Device Certificate column displays Valid for each firewall
  • The Device Certificate Expiry Date shows a future date
▶ 4.3 Restoring an Expired Certificate

If the certificate shows Expired:

  • Standalone devices: Generate a new OTP and repeat Section 4.1.
  • Panorama-managed devices: Repeat Section 4.2 — generate a new OTP Request Token, get a new OTP from CSP, and upload it.
  • TPM devices (PA-400, PA-1400, PA-3400, PA-5400 series): Use CLI without OTP:
admin> request certificate fetch
▶ 4.4 Post-Certificate: WildFire Users

If using WildFire or Advanced WildFire, refresh the registration channel on each firewall (standalone or Panorama-managed):

admin> request wildfire registration channel public

Step 4 Checklist

Verify Before Proceeding

Standalone firewalls:

  • OTP generated from CSP (60-minute window, one attempt)
  • NTP configured and committed on each firewall
  • Device certificate installed successfully
  • Certificate status verified as Valid

Panorama-managed firewalls:

  • Panorama Superuser access confirmed
  • Managed firewalls on the same CSP account as Panorama
  • NTP configured on Panorama and pushed to managed firewalls via template
  • OTP Request Token generated from Panorama
  • OTP generated from CSP using “Panorama managed firewalls” option
  • OTP uploaded via Panorama (pasted, not file upload)

Both:

  • (WildFire users) Registration channel refreshed on each firewall

Step 5: VM-Series Specific Prerequisites

VM-Series Only

This step applies only to VM-Series firewalls. Hardware NGFW users can skip to the Pre-Flight Checklist.

VM-Series firewalls follow a different onboarding path than hardware NGFWs. Before onboarding to SCM, you must activate your credit pool, create deployment profiles, and ensure your cloud platform allows management-plane connectivity.

5.1 Licensing Model Comparison

Understand which licensing model applies to your VM-Series deployment.

Aspect Software NGFW Credits BYOL (Fixed Model)
Flexibility Adjust vCPUs and services anytime Fixed at purchase
PAN-OS 10.0.4+ (max 64 flexible vCPUs) All versions
Panorama Cost Included (vCPU cost only) Separate purchase
Security Services Add/remove via deployment profile Bundle 1 or Bundle 2 (fixed)
Credit Reuse Credits return to pool when VM deactivated Non-reusable
Registration Automatic via auth code Separate manual step
SCM Deployment Profile Required — created in CSP Not applicable
Which Model Am I Using?

If you purchased a credit pool and create deployment profiles with auth codes starting with “D,” you are using Software NGFW Credits. If you purchased a specific VM model license (VM-200, VM-300, etc.) with a traditional auth code, you are using BYOL.

5.2 Activate Software NGFW Credit Pool

If using Software NGFW Credits, your credit pool must be activated before creating deployment profiles.

  1. Check for the activation email from Palo Alto Networks containing:
    • Credit pool ID
    • Start and end dates (term: 1–5 years)
    • Total credit amount
  2. Click the activation link or log into CSP.
  3. Navigate to Products > Software NGFW Credits > Start Activation.
  4. Select the credit pool to activate.
  5. Select the support account to associate with.
  6. Choose a default credit pool for deposits.
  7. Click Deposit Credits to complete activation.
Credit Expiration

Both allocated and unallocated credits expire at the end of the term. If a firewall is deactivated while it has internet access, its credits are refunded to the pool for reallocation. Plan your deployments accordingly.

5.3 Verify Deployment Profile Status

You should have already created and activated your deployment profile in Section 2.3. Before continuing, confirm:

  • Profile Association Status shows Complete (not “Associating” or “Pending”) in Common Services > Tenant Management > Deployment Profiles
  • If the status is still “Associating,” wait up to 50 minutes for provisioning to finish

5.4 Generate Auto-Registration PIN

The auto-registration PIN enables automatic device certificate installation during VM-Series bootstrap, eliminating the need for manual OTP-based certificate fetch (Step 4).

  1. Log into CSP.
  2. Navigate to Products > Device Certificates.
  3. Generate a Registration PIN for your VM-Series deployment.
  4. Record both values:
    • PIN ID (e.g., abcdefgh1234****)
    • PIN Value (e.g., zyxwvut-0987****)

These values are used in the bootstrap configuration (init-cfg.txt or cloud user data):

# In init-cfg.txt or cloud instance user data:
vm-series-auto-registration-pin-id=abcdefgh1234****
vm-series-auto-registration-pin-value=zyxwvut-0987****
PIN vs OTP

The auto-registration PIN is different from the OTP used in Step 4. The PIN is reusable across multiple VM-Series deployments and installs the device certificate automatically during bootstrap. The OTP is single-use (one attempt, 60-minute window) and requires manual installation. For VM-Series, the PIN method is preferred.

5.5 Cloud Platform Readiness

Your cloud environment must allow the VM-Series management interface to reach Palo Alto Networks cloud services. This is in addition to the general network requirements in Step 3.

▶ AWS Requirements
  • Management subnet — must have internet access (via Internet Gateway or NAT Gateway)
  • Security Group on management interface — allow outbound on all ports from Step 3, Section 3.1
  • IAM Instance Profile — required if using bootstrap from S3 (s3:GetObject, s3:ListBucket permissions)
  • User Data / S3 Bootstrap — include vm-series-auto-registration-pin-id and vm-series-auto-registration-pin-value in init-cfg.txt
▶ Azure Requirements
  • Management subnet — must have internet access (public IP or NAT Gateway on management NIC)
  • NSG on management subnet — allow outbound on all ports from Step 3, Section 3.1
  • Custom Data — include PIN ID and PIN Value in the VM’s custom data field during deployment
  • Managed Identity — required if using bootstrap from Azure File Share
▶ GCP Requirements
  • Management subnet — must have internet access (external IP or Cloud NAT on management NIC)
  • VPC Firewall Rules — allow egress on all ports from Step 3, Section 3.1
  • Instance Metadata — include PIN ID and PIN Value as custom metadata key-value pairs
  • Service Account — required if using bootstrap from GCS (storage.objects.get, storage.objects.list permissions)
▶ Other Platforms (ESXi, Hyper-V, KVM, OCI)
  • Management network — must have routable internet access to Palo Alto Networks cloud services
  • Firewall rules — allow outbound on all ports from Step 3, Section 3.1 from the management interface
  • Bootstrap ISO / USB — include init-cfg.txt with PIN ID and PIN Value
  • DNS and NTP — must be reachable from the management interface (see Step 3, Section 3.8)

5.6 Configure “Managed By” Setting

Before onboarding a VM-Series firewall to SCM, set the management mode on the device.

  1. Log into the firewall web interface.
  2. Navigate to Device > Setup > Management.
  3. Under Panorama Settings, select Managed By Cloud Service.
  4. Set the connection port: 3978 (default) or 443.
  5. (Optional) Enable Compress Config for large configurations.
  6. Commit.
Important

Changing management mode to “Cloud Service” is a prerequisite for SCM onboarding. This is configured on the firewall, not in SCM.

Step 5 Checklist (VM-Series Only)

Verify Before Proceeding

Software NGFW Credits:

  • Credit pool activated — credits deposited and available in CSP
  • Deployment profile — created (Section 2.3), auth code (starts with “D”) recorded
  • Profile association status — shows “Complete” (if cloud subscriptions included)
  • Auto-registration PIN generated — PIN ID and PIN Value recorded

BYOL:

  • VM model license activated — auth code registered in CSP
  • Device certificate installed — via OTP (Step 4) since BYOL does not use auto-registration PINs

All VM-Series:

  • Cloud platform — management subnet has internet access
  • Security groups / firewall rules — outbound ports from Step 3 allowed from management interface
  • Bootstrap config — init-cfg.txt or user data includes PIN ID/Value (Software NGFW) or auth code (BYOL)
  • “Managed By Cloud Service” — configured on each firewall
  • Connection port — 3978 or 443 set consistently

Step 6: Onboard Devices to Strata Cloud Manager

6.1 NGFW Onboarding

This workflow applies to both hardware NGFWs and VM-Series firewalls.

Step 1: Configure the Firewall Locally

  1. Verify DNS and NTP are configured under Device > Setup > Services.
  2. Navigate to Device > Setup > Management > Panorama Settings.
  3. Select Managed By Cloud Service.
  4. (PAN-OS 11.2+) Choose communication port: Default (dedicated port 3978) or 443 (standard SSL).
  5. (Optional) Enable Compress Config for faster configuration transfers.
  6. Commit changes.
Port Selection (PAN-OS 11.2+)

PAN-OS 11.2 and later allow you to choose between the dedicated management port (3978) and standard SSL (443). Use port 443 if your network restricts non-standard ports. Older PAN-OS versions always use port 3978.

Step 2: Associate Devices in SCM

  1. In SCM, go to System Settings > Device Associations > Add Devices.
  2. Select the serial numbers of firewalls to onboard.
  3. Associate each firewall with Strata Cloud Manager or AIOps for NGFW.

Step 3: Move to Cloud Management

  1. Go to System Settings > Device Management > Available Devices.
  2. Select the firewall(s).
  3. Click Move to Cloud Management.
Important

Moving a device to cloud management transfers configuration authority from the local firewall (or Panorama) to SCM. Local configuration changes will be overridden by SCM on the next push. Ensure your configuration is backed up before proceeding.

Step 4: Verify

  1. In SCM, check Cloud Managed Devices — status should show Success.
  2. On the firewall CLI, run:
admin> show cloud-management-status

Confirm Connected shows Yes.

Expected Output

The show cloud-management-status command should show:

  • Connected: yes
  • Endpoint FQDN: your regional SCM endpoint
  • TCP status: Connected
  • DNS status: Resolved

7.2 Device Labels & Onboarding Rules (At Scale)

For environments with many devices, automate onboarding with labels and rules.

Device Labels

  1. Navigate to Configuration > NGFW and Prisma Access > Setup > Device Onboarding > Device Labels.
  2. Create labels to group devices by software version, location, role, etc.
  3. Apply labels to devices during or after onboarding.

Onboarding Rules

Onboarding rules automatically assign configurations to devices as they connect to SCM.

Match Criteria Description
Model Match by firewall hardware model (e.g., PA-460, VM-300)
Serial Number Match by serial number pattern (supports regex)
Labels Match by previously assigned device labels
Action Description
Target Folder Auto-assign the device to a specific SCM folder
Snippets Apply standardized base configurations (security policies, profiles, etc.)
VPN Onboarding Automatically configure Auto VPN clusters
User Context Onboarding Integrate with Cloud Identity Engine for user-based policies
Snippets

Snippets are reusable configuration blocks that can be applied to multiple folders and devices. Create a base security snippet with your standard policies, then reference it in onboarding rules to ensure consistent configuration across all new devices.

7.3 Post-Onboarding Configuration in SCM

Once devices are onboarded, configure them through SCM.

General Settings

  • Domain name, login banners, certificates, time zones
  • Tunnel acceleration for improved VPN performance
  • Session settings — VPN session timeouts and parameters

Folder Management

Organize firewalls into folders for group configuration.

HA Pair Requirement

HA (High Availability) pairs must be in the same folder. Placing HA peers in different folders will cause configuration push failures.

Interface Assignment

SCM provides predefined interface variables for consistent configuration across devices:

Variable Default Interface Purpose
$eth-internet eth1/3 Outbound / internet-facing traffic
$eth-local eth1/4 Local / internal network connections

Verify or customize these assignments to match your physical interface topology.

Push Configuration

Important

Changes made in SCM do not take effect on devices until you perform a Push Config from the Configuration menu. Always push after making changes to apply them to managed devices.

Troubleshooting

Certificate Fetch Issues

Symptom Likely Cause Resolution
“Failed to fetch device certificate” OTP expired (>60 min) or already used Generate a new OTP from CSP. Each OTP allows only one attempt.
Certificate fetch times out Management interface MTU too high Lower the management interface MTU to 1374: set deviceconfig system setting mtu 1374 then commit and retry.
Certificate validation fails System clock is wrong Verify NTP is configured and synced: show ntp. Fix NTP and commit before retrying.
“TPM public key match failed” Backend mismatch on TPM models Open a TAC case — the certificate record may need to be reset.
otp keyword not available in CLI Firewall has a TPM TPM devices don’t need an OTP. Run: request certificate fetch
Cert fetch fails through proxy Proxy blocking required ports Ensure proxy allows traffic to certificate.paloaltonetworks.com and api.paloaltonetworks.com on port 443.
Panorama OTP Upload fails Pasted file content or OTP expired You must paste the OTP text. Generate a new OTP if expired.

VM-Series Specific Issues

Symptom Likely Cause Resolution
Auto-registration PIN fails during bootstrap PIN ID/Value incorrect or management interface has no internet Verify PIN values in init-cfg.txt. Confirm management subnet has internet access (check security group/NSG/firewall rules).
Deployment profile shows “Associating” indefinitely Tenant provisioning stalled Wait up to 50 minutes. If still stuck, contact Palo Alto Networks support.
Credits not available after activation Credits deposited to wrong pool or wrong CSP account In CSP, verify the credit pool under Software NGFW Credits. Ensure the correct support account is selected.
Firewall not appearing in SCM after bootstrap “Managed By Cloud Service” not set, or certificate not installed Log into the firewall → Device > Setup > Management → verify “Managed By Cloud Service” is selected. Check certificate status with show device-certificate status.
Auth code “D” not accepted Using auth code in wrong field or profile not activated Auth codes starting with “D” are for deployment profiles, not traditional licensing. Ensure the profile is activated and associated with a tenant.

Connectivity Validation

What to Validate How to Test Expected Result
DNS resolution ping host ds-cloudmgmt.paloaltonetworks.com Resolves to an IP address
Certificate FQDNs ping host certificate.paloaltonetworks.com Resolves to an IP address
NTP sync show ntp Status shows “synced”
License status request license info Shows active licenses

Cloud Management Status: Not Connected

Symptom Likely Cause Resolution
Connected: no — TCP status “Failed” or “Connection timed out” Port 3978 (or 443 on PAN-OS 11.2+) blocked outbound Verify firewall can reach the SCM endpoint on the correct port. Check transit firewall rules, proxies, and NAT.
Connected: no — DNS failure in status output Firewall cannot resolve SCM FQDNs Verify DNS servers under Device > Setup > Services. Test with ping host ds-cloudmgmt.paloaltonetworks.com.
Connected: no — endpoint FQDN is empty Device not properly onboarded in SCM Verify the device serial number is associated in SCM > System Settings > Device Associations and moved to cloud management.
“Device Not Found” in SCM Device not associated with correct Tenant Service Group Verify serial number is registered and associated with the correct tenant in the Palo Alto Networks Hub/CSP.
Status shows “Onboarded” but “Disconnected” Device certificate missing/expired, or port 3978 blocked Check certificate status (show device-certificate status) and verify port connectivity.
Restart Cloud Management Process

If everything looks correct but the device still won’t connect, restart the cloud management process and wait 5 minutes:

admin> debug software restart process cloud_mgr

Telemetry Not Appearing in SCM Dashboard

Symptom Likely Cause Resolution
No telemetry data after 2+ hours Telemetry not enabled on the device Check with show device-telemetry details. For PAN-OS versions below auto-enable thresholds, manually enable telemetry sharing.
Telemetry enabled but data missing Region mismatch between telemetry and SCM/CDL instance Check telemetry region: show device-telemetry settings. Ensure it matches your SCM instance region.
Telemetry intermittent or stale Ports 5222–5224, 5228, 5229 blocked Verify these telemetry ports are open outbound in addition to 443.
Telemetry stuck — “cannot lock /etc/subgid” in logs Known bug in some PAN-OS 10.2.x versions On PAN-OS 10.2.11+: delete authentication system-lock-files. On older versions: restart management plane or contact TAC.

Configuration Push Fails After Onboarding

Symptom Likely Cause Resolution
Push fails immediately Configuration conflict with local settings Check SCM > Configuration > Operations > Push Status for specific error details. Resolve conflicts between SCM config and local firewall config.
Push hangs or times out Connectivity issue between SCM and firewall Verify show cloud-management-status shows Connected. Check for packet loss on the management path.
HA peer push fails HA peers in different SCM folders Move both HA peers to the same folder in SCM. HA pairs must always be in the same folder.
▶ Service Route Configuration (if management interface lacks internet)

If your firewall’s management interface does not have internet access (common in security-hardened environments), configure a service route so cloud traffic uses a data-plane interface instead.

  1. Navigate to Device > Setup > Services > Service Route Configuration.
  2. Select Customize.
  3. For Palo Alto Networks Services (or specifically Cortex Data Lake), assign a data-plane interface/IP that has internet access.
  4. Commit and verify connectivity.
